## Key design decisions

### Why plain Vite+Svelte, not SvelteKit?

SvelteKit adds SSR, routing, and server-side concepts that are unnecessary
(and slightly heavier) for a desktop app with a single-window layout.
Plain Vite+Svelte keeps the build output small and the mental model simple.

The `$lib` path alias is manually configured in `vite.config.ts` and
`tsconfig.json` to mirror SvelteKit's convention without requiring it.

### Why one `stores/` folder instead of one store per feature?

Each file in `stores/` owns one domain (library, launcher, config). Within
each file, related state and actions live together. This avoids the
anti-pattern of scattering related state across many tiny files while still
keeping clear boundaries between domains.

### Why `RunningGames(Arc<Mutex<HashMap<...>>>)` instead of Tauri's `State`?

Tauri's `State<T>` IS the mechanism — `RunningGames` wraps the actual data in
`Arc<Mutex<_>>` so it can be safely shared across multiple async Tauri
commands without blocking the main thread. The `Arc` is needed because Tauri
commands run in a thread pool.

### Why not use `tauri-plugin-process` for spawning?

The plugin provides basic `Command` spawning but does not expose the level of
environment variable control GPTK requires (`DYLD_LIBRARY_PATH`,
`WINEDLLOVERRIDES`, etc.). Using `std::process::Command` directly gives full
control while remaining straightforward.

## Adding a new Tauri command

1. Write the function in the appropriate module (`config.rs`, `launcher.rs`,
   `steam.rs`) or directly in `main.rs` for simple one-liners.
2. Annotate with `#[tauri::command]`.
3. Add it to `tauri::generate_handler![]` in `main.rs`.
4. Add a corresponding `invoke<ReturnType>("command_name", args)` call in the
   appropriate store or component.
5. Add a TypeScript type if the return type is new.
6. Document it in `docs/API.md`.

## Adding a new Svelte component

1. Create `src/lib/components/MyComponent.svelte`.
2. Import from `"../types"` and `"../stores/..."` using relative paths inside
   `src/lib/`, or from `"$lib/..."` from `src/` level files (the alias is
   configured in `vite.config.ts`).
3. Keep component state local; push anything shared into a store.
